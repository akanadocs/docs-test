Administrator Accounts

    There are three different and distinct administrator roles that can be assigned to user accounts within the application. All three roles are described here in order to provide context for the Installation Administrator responsibilities and required skills as described in this document.

Installation Administrator
    This System Admin Guide is written and intended for the users assigned the role of Installation Administrator, also known as the system administrator for this installation. See System Admin Console for information on the UI provided to Installation Administrators. The initial list of Installation Admins is set in the lib.conf file in the {admin.users} property. After the initial installation, the list is maintained by the existing Installation Administrators through use of the SetInstallationProperty command. See the section Update Installation-wide Properties for more details.

    Responsibilities of the Installation Administrator generally include these types of activities:

    • Coordinate installation of the base product and service packs with the underlying server sysadmins

    • Creating the initial Asset Library and additional libraries as required

    • Obtaining and reviewing system logs

Library Administrator (also known as the Library Support Account)
    Each Asset Library is created with a user account that is authorized to all library administrative roles. This account is designated as the Library Administrator or Library Support Account.

    Responsibilities generally include these types of activities:

    • Managing the Library parameters that are configurable (post-Library creation)

    • Managing the Global Definition Template used for the library's Asset definition

    • Managing the Library's UDDI publishing

    Please see the help system available in your Portfolio Manager or Lifecycle Manager web browser application for more information on the Library Administrator role.
Usage Controller
    The Usage Controller role is assigned to someone in the library who works to provide the necessary group structure, role assignment and template associations to accommodate the processes established for asset production and consumption activities.

    Each Library has one or more users assigned the role of Usage controller.

    The responsibilities generally include these types of activities:

    • Create and manage the library group and project hierarchy

    • Manage asset capture (constraint) templates

    • Assign and manage user roles and authorization

    • Create and manage additional items used in the library configuration, such as group profiles, asset views and classification criteria sets.

    Please see the help system available in your Portfolio Manager or Lifecycle Manager web browser application for more information on the Library Administrator role.
Project Manager
 

A Project Manager can perform user management tasks for your library, much like users assigned the Usage Controller role described above. However, the Project Manager role is assigned by project, and the capabilities of the Project Manager is scoped to their own project only.

Each project in a library has one or more users assigned the role of Project Manager.

The responsibilities generally include these types of activities:

  • Create new library users and assign them to the project. This capability can be withheld.

  • Manage group role assignments for project users.

  • Approve asset reuse requests instigated by project users. This capability is optional.

Visible Asset Sources

Each library consists of two components: the library itself, which contains the published assets, and its associated asset source, where assets are created and edited prior to being published. When assets are published from an asset source into its associated library, you can also designate that a simultaneous publish take place into other libraries as well. The configuration necessary to cause this to take place occurs in the target library and consists of the establishment of a Visible Asset Source for each source library desired. Libraries that share content in this fashion are said to be "federated". Note that content shared in this manner can include reference models as well as assets. Published assets are refreshed in all applicable libraries with each publish from the originating asset source. In cases where the Visible Asset Source entry uses a remote library, the update takes place according to the polling interval established by the Remote Asset Source entry used by the Visible Asset Source.

Create a Visible Asset Source

On the Manage Visible Asset Sources page, selecting your library from the Library drop down list causes the page to refresh and display a list of all other libraries existing for your installation. If you want to pull assets and models from a visible library into your library, click on the corresponding Selected box. As you do so, there are two important things to consider and choices to make:

  1. Ensure that you select a Responsible User that corresponds to a user account in your library that is associated with the "Reporting Group" you intend. The responsible user's reporting group will:
    1. Be used as the owning group for the assets from this asset source.
    2. Designate the publish template(s) to be used for the assets published from this asset source. The publish template used for these assets can:
      1. Provide publish information, such as designation of private artifacts and establishment of asset owner involvement in asset reuse approvals.
      2. Automatically establish forum topics for the assets' discussion forums.
    3. Note: the default "Responsible User" account is the account called Portfolio Manager or Lifecycle Manager. This is a user account associated with the application itself, not a human user. The Super user account is configured in your web browser application in a fashion similar to other user accounts.
  2. Establish a Classification Criteria Set to use as a stereotype in order to filter the assets published from the Visible Asset Source. By naming a classification criteria set, you cause only those assets that meet the classification criteria of the named set to be included in the publish. Classification Criteria Sets (CCSs) are created and managed by your library's Usage Controller(s) through your web browser application.

See the help system available online in the application for more information on: Classification Criteria Sets, Publish Templates and their association with groups and asset types, asset publish information, Reporting Groups, and Reference Models.

Setting up LDAP

    All of the configuration parameters needed to setup Portfolio Manager or Lifecycle Manager against a preexisting LDAP server or a new server are in the configuration file.

    There are two distinct modes Portfolio Manager or Lifecycle Manager can operate in with respect to an LDAP server: native and guest mode. The first step in setting up the application to use an LDAP server is to determine which of these modes to use.

    • guest: Guest mode provides a company the ability to configure the application against a preexisting LDAP server. Most likely the LDAP server will already have a set of users in the directory. Portfolio Manager or Manager Lifecycle Manager will only use this LDAP server to search for users and authenticate them. It will not perform any updates to the directory.
    • native: In this mode we assume that an existing LDAP server is not available. In this case you can install an OpenLDAP server (freely downloadable) and configure Portfolio Manager or Lifecycle Manager to use it. Management of users in the LDAP server is transparently done through the library pages. See the installation guide on how to install OpenLDAP.

    The following parameters describe how to configure the installation properties. In most cases the defaults will be acceptable. However: if you plan to use 'guest' mode, then you will need to know a few details about your existing LDAP server. Each property is described differently depending on what mode is configured. See also the Installation Properties section.

    • ldap.ownership.mode: Either set to 'native' or 'guest' as described above

    • ldap.binddn: This is the LDAP Distinguished Name if the LDAP server requires authentication to perform searches or modify entries.
      Guest: Set this to the distinguished name of such a user, if required.
      Native: This does not need to be set as a default value of "cn=Manager,ou=People,dc=Logidex" will be used.

    • ldap.bindpassword: This only needs to be set if the ldap.binddn property is set. The bind password is the password associated with the distinguished name, described above.
      Guest: If ldap.binddn is set, set this to the associated password, otherwise leave blank.
      Native: This does not need to be set as a default value of whatever logidex.password is set to will be used. You can however set this to a different value for security purposes.

    • ldap.follow.referrals: Either set to 'true' or 'false'. LDAP referrals are a way of directing LDAP actions to other servers in the event the target does not have the information requested. Following referrals requires additional connection overhead and is only recommended if you know that your company uses referrals.
      Guest: Only set this to true if you require the ability to find users located on different LDAP servers.
      Native: This does not need to be set as a default value of false will be used. Setting this to true will only incur a performance penalty.

    • ldap.url: This is a LDAP URL which conforms to RFC2255, with the ability to specify 'ldaps' as a protocol. This URL has two main purposes, first it specifies what host the LDAP server is on. Secondly, the URL contains information on how to find users. The URL is broken into several parts: protocol, hostname, port, base distinguished name(DN), attributes, search scope, filter, and extensions. At a minimum a hostname and base DN must be specified. Other parameters may be necessary depending on your configuration. The URL takes the format: protocol://hostname:port/baseDN?attributes?scope?filter?extensions. An example would look like (ldap://localhost:389/ou=People,dc=Logidex?uid?one??). Trailing question marks may be omitted.
      Guest: This is highly dependent on the LDAP schema, configuration, and company setup. Read through the following set of configuration points to determine how to set up this property.
      Native: This should not need to be set as a default value of 'ldap://localhost:389/ou=People,dc=Logidex?uid?one??' will be used. See the examples below to change the port number OpenLDAP runs on.

      • protocol: This can be set either to 'ldap' or 'ldaps'. If set to 'ldap' all ldap communications will be sent unencrypted. If the protocol is 'ldaps' then communications will be done with TLS/SSL 3.0. This requires setting up certificates on the LDAP server and configuring the application server as well, which is beyond the scope of this section.

      • hostname: This can either be the hostname (preferred) or an IP of the LDAP server.

      • port: If the port is not specified, it will default to 389 for ldap, and 636 for ldaps.

      • base DN: This is the distinguished name of where to start LDAP searches. To enhance performance, you should set this as deep as possible in the LDAP Directory Information Tree to maintain the smallest set of users you want to be visible.

      • attributes: This is a comma separated list of LDAP attributes. Only one attribute should be supplied here. If it is omitted, then a default attribute of 'uid' will be used. This is the attribute used to look up users based on a login name. Most commonly used to distinguish users are the 'cn' (common name) or 'uid' (user id) attribute. The distinguished names of the users will probably determine what to specify here. For example, if most of the users in the LDAP directory look like 'cn=John Doe, ou=unit, dc=...' then you should specify 'cn' as the attribute to use to search. Note: if more than one entry for a given user exists (even though they may logically refer to the same entity), authentication will be denied as it does not know which to use.

      • search scope: This can take one of three values: 'base', 'one', or 'sub'. The scope determines how to should search the entries in the LDAP server trying to find a user. Base simply means only search the entry referred to in the base DN. Since this never returns more than one result it should never be used. A value of 'one' means only search the direct descendants of the base DN. A value of 'sub' signifies that the base DN along with all the subordinates of the base DN should be searched. The default used if a scope is not specified is 'one'.

      • filter: This is an LDAP filter used to limit the number of users allowed access. A user is only allowed access if they meet the previous requirements to login, plus pass the filter. LDAP filters are rather complex and not discussed here. As an example, specifying: '(employeeNumber=*)' would only allow users that have an employee number in their LDAP entry to login. Before limiting users this way, make sure you successfully configure authentication against LDAP without filters, only then should you use the filtering mechanism.

      • extensions: Extensions are not used at this time. This field will most likely be blank.

    If any of the following characters appears as part of the name or value of an RDN they must be escaped with a backslash: 

    , + " \ < > ;

    Also, due to fact that the lib.conf also interprets backslashes, one must double escape any backslashes. For example: ldap.binddn=cn=John Doe\\, Jr.,ou=users,dc=example,dc=com

    ldap.url - LDAP server notes and examples:

    OpenLDAP - native mode - ldap://localhost:1234/ou=People,dc=Logidex?uid?one
    This URL should only be used if in native mode. It simply uses port 1234 instead of the default.
     
    Windows Domain Controller -
    ldap.url=ldap://windows-dc.example.com/CN=Users,DC=example,DC=com?sAMAccountName?one??
    One of the following
    • ldap.binddn=CN=normaluser,CN=Users,DC=example,DC=com (LDAP format)
    • ldap.binddn=normaluser@example.com (Windows 2000 and later account format)
    • ldap.binddn=example\\normaluser (pre-Windows 2000 account format, note the double '\')
    ldap.bindpassword=password
    This is the format used to connect to a Windows Domain Controller for authentication. The hostname is that of the domain controller. The base DN (CN=Users,DC=example,DC=com) is obtained taking 'CN=Users' and appending each component of the domain beginning with 'DC='. The attribute (sAMAccountName) is the Windows Active Directory (LDAP) attribute used for authentication. Another possibility includes using the common name attribute: 'cn'. The ldap.binddn for Windows AD is simply a named user prepended with 'CN=' and appended with the binddn. The ldap.bindpassword is simply the password the user associated with ldap.binddn uses to login to Windows. The user and password should be a user with minimal privileges and one that isn't normally associated with a human. You can use the Windows Active Directory management console to determine the actual DN and attribute for authentication.
     
    SunONE/iPlanet Directory Server - ldap.url=ldap://iplanetserver.example.com/OU=People,DC=example,DC=com?uid?one
    In this example, the LDAP server is located on a host called iplanetserver.example.com, running on the default port of 389. The base DN (OU=people,DC=example,DC=com) is obtained by taking 'OU=People' and appending each component of the domain beginning with 'DC='. The attribute (uid) is what the iPlanet Directory Server uses as the default identifier. You can use the administrative console to determine the exact DN and attribute. Depending on the setup, you may have to specify an ldap.binddn and ldap.bindpassword if the LDAP server is not setup for anonymous searching.
     

Setting up LDAP over SSL

    It is best to make sure the installation works without SSL first, then turn on SSL after you have verified that all the configuration parameters are set correctly. The steps to setup LDAP over SSL are as follows and will depend on the type of LDAP and application server you have installed.

    1. Configure and setup the LDAP server with a certificate to use for SSL
    2. Export the certificate and save it in DER format (other formats may work).
    3. Import the certificate into the keystore for the JDK the application server is using
    4. Update the lib.conf file by changing the ldap.url to ldaps://servername:636/...
      Run ApplicationRoot/install update-config
    5. Restart the Application server's JVM

     

    Here is a step-by-step example for Novell eDirectory and WebLogic:

  • Configure and setup the LDAP server with a certificate to use for SSL
    • In ConsoleOne (NDS's admin console), click on the LDAP container in the left nav tree that contains the server you wish to use SSL for. The right side should contain an "LDAP Server - servername" entry
    • Right click in the right hand pane (somewhere in the unused space) and select New -> Object; select "NDSPKI:Key Material"
    • Step through the wizard, the defaults should be good enough, but ensure the server is correct. Give it a meaningful name like "LDAP SSL".
    • Right click on the "LDAP Server" entry and select properties.
    • Select the SSL/TLS Configuration tab
    • For the server certificate select the SSL certificate you just created and change any options you'd like.
    • Select OK
  • Export the certificate and save it in DER format (other formats may work).
    • Right click on the certificate you just created (e.g. "LDAP SSL - ldapserver") and select properties
    • Click the Certificates tab and make sure the tab's dropdown has the "Trusted Root Certificate" selected, and then select Export
    • Export the certificate in DER format
  • Import the certificate into the keystore for the JDK the application server is using
    • Login to the application server (repeat for each server in the cluster if applicable)
    • Copy the exported certificate from above to the machine
    • Add the key to the java keystore using the keytool command, e.g.:
      $JAVA_HOME/bin/keytool -import -v -alias NDSSSL -file /tmp/TrustedRootCert.der -keystore $JAVA_HOME/jre/lib/security/cacerts
  • Update the lib.conf file
    • ApplicationRoot/bin/libctl stop appserver
    • Change ldap.url to use SSL - ldap.url=ldaps://...
    • install update-config
    • ApplicationRoot/bin/libctl start appserver
  • Restart the App Server's JVM
    • stop WebLogic
    • start WebLogic

Setting up SSL

    Portfolio Manager and Lifecycle Manager support the use of SSL (Secure Sockets Layer) encryption. However, by default, the installation is not configured to use SSL. If you want to use SSL for your installation, you will need to modify the default configuration. Changes are required on both the Web Server and the App Server.

    Update the Web Server

    These updates are for an Apache web server.

    1. Login to the web server as root.
    2. Create and Install SSL Certificates
      If you will be using SSL on your web server, you will need to request your digital certificates from a certificate authority (CA), or you can self-sign the certificate. Below are instructions from two CAs - Verisign and GeoTrust (formerly Equifax) on requesting a certificate:
      http://www.verisign.com/support/csr/apache/v00.html
      http://www.geotrust.com/quickssl/install/index.htm

      NOTE - the SSL certificate host name must match the {web.server.host} property in the lib.conf file.

    3. Copy the certificate, key, and CA file (if applicable) to: 
        ApplicationRoot/httpd/conf/your_company.com.crt
  ApplicationRoot/httpd/conf/your_company.com.key
  ApplicationRoot/httpd/conf/some_CA.crt
    4. Modify the Apache configuration to add a SSL virtual host: 
        $ vi /usr/local/apache/conf/httpd.conf  (or any editor) 
  
  <IfDefine HAVE_SSL>
  <VirtualHost _default_:443>
    SSLEngine on
    SSLCipherSuite ALL:!ADH:!EXPORT56:RC4+RSA:+HIGH:+MEDIUM:+LOW:+SSLv2:+EXP:+eNULL
    
    SSLCertificateFile conf/your_company.crt
    SSLCertificateKeyFile conf/your_company.key
    SSLCACertificateFile conf/some_CA.crt         
    SetEnvIf User-Agent ".*MSIE.*" \
         nokeepalive ssl-unclean-shutdown \
         downgrade-1.0 force-response-1.0
  </VirtualHost>
  </IfDefine>
                 
  Add the above section and update the lines for the SSL certificates to refer
  to the certificate, key, and CA files from above
    5. Modify the automatic startup shell script 
        Solaris
    vi ApplicationRoot/httpd/bin/apache-init.d
    
    Change "$CONTROL start" to "$CONTROL startssl"
    6. Restart Apache 
        ApplicationRoot/bin/apachectl stop
  ApplicationRoot/bin/apachectl startssl
    Update the App Server
    1. Update the lib.conf file or use the SetInstalltionProperty command to change the usessl setting to one of these values:
      login
      Use SSL only on the login screen (where passwords are used)
      all
      Use SSL for all web pages
      none
      Do not use SSL on any web pages
    2. Apply the changes and restart the App Server

Log Files and Logging Levels

Log files and location

Below is a brief description of the most important log files (not all of the log files are listed here).

By default the log files are in the directory ApplicationRoot/logs, with subdirectories (listed below) for each major component.

  • webapp
    app_N.log
    This log file will contain a history of the transactions that have gone thru the Portfolio Manager or Lifecycle Manager library.
    incident_NN.log
    This log file is used to capture error exceptions (called incidents) that are generated in the library. In almost all cases, this log will contain lower level detail that will pinpoint the cause of the problem. It may be necessary to call the SOA Software support team to interpret this log file.
    startwl-$date.log and stopwl-$date.log (WebLogic)
    This log captures stdout when using libctl to start WebLogic.
    startjboss-$date.log(JBoss)
    This log captures stdout when using libctl to start JBoss.
  • appserv
    The appserv directory provides a great deal of detail about the operational aspects of the application server product itself. It provides useful information to verify that the server has started correctly and with the correct parameters. There may be times when there is additional information in this log that will be needed for debugging. This directory should be a symlink to the application server's logging directory.

com.logiclibrary.common.logging.ThreadFileHandler.pattern=/opt/lm/logs/webapp/lm_%g.log

com.logiclibrary.common.logging.FileHandler.pattern=/opt/lm/logs/webapp/lm_%g.log
The only parameters that should need to be changed are the pathnames where the log files are placed and the logging level. The *.pattern properties specify where to place individual log files and how to number them; the %g is replaced with a sequential number. Once a log reaches the limit (*.limit property) the log number is increased. Once the log number reaches the *.count property the count is reverted to 0 and overwrites any previous log file with that name.

Possible logging levels are as follows (from most verbose to no logging):

  • ALL
  • FINEST
  • FINER
  • FINE
  • CONFIG
  • INFO
  • WARNING
  • SEVERE
  • OFF

The default level for new installations is FINER, and can be changed to ALL or FINEST to get more detailed debugging information.

NOTE: Setting the logging level to one of the more "verbose" levels will tend to slow down the performance of the application so you should only turn up the logging if needed for debugging.

Super User

Portfolio Manager or Lifecycle Manager provide the capability of accessing your libraries and your System Admin console by logging in with a special "user" account that does not require authentication by your LDAP system. This account, termed the "Super User" account can be used via the web browser application to access your library and the System Admin console. Logging in as Super User provides you with unlimited access to the libraries of your installation and the facilities provided by the System Admin console. Note that super user access can be enabled or disabled in the lib.conf file by assigning a password or leaving the password blank.

The Purpose of Super User

Super User access is provided to enable you to recover from the lack of an active Library Administrator account. By logging into a library as super user, you can activate an inactive account or create a new user account and assign the Library Administrator role.

Another purpose for the Super User account is to provide you with a user account that is portrayed as being associated with the application as opposed to being associated with an individual, human user. Portfolio Manager or Lifecycle Manager provides for the use of the super user account as an "application user" called Portfolio Manager Application User or Lifecycle Manager Application User. This user is available for such things as being recorded as the "submitter" of assets submitted by automation and being identified as the "responsible user" when establishing library federation through creation of Visible Asset Source and Remote Asset Source configuration.

Super User Role Assignments

The Super User account is assigned the library roles of Library Administrator, Usage Controller and Asset User. In addition, this account is also assigned the group roles of Asset Capture Engineer (ACE), Publisher and Asset Owner - all at the Enterprise Group level. See the web browser application's online help for more information on roles and their scope of application.

    Customize user attributes and LDAP attributes

      The user properties that appear in the create and display user and contact pages are obtained and configured through the attribute definitions. Attribute definitions determine not only the names of fields that are available and displayed for users, but also the corresponding services to use when looking up user information. This is accomplished by configuring an XML document and uploading it through the Admin Console using the Attribute Definition commands: GetAttributeDefinitionSchema, GetCurrentAttributeDefinitions, GetDefaultAttributeDefinitions, and SetAttributeDefinitions.

      The following is an example document. 

      <?xml version="1.0" encoding="utf-8"?>
<ContactAttributeDefinitions xmlns="http://www.logiclibrary.com/2004/ContactAttributesXML">
        <AttributeDefinition attribute-name="Full Name" include-in-filter="true" include-in-list="true">
    <ExternalSourceMapping source-id="LDAP" name-in-source="cn" />
  </AttributeDefinition>
  <AttributeDefinition attribute-name="Email" include-in-filter="true" include-in-list="true">
    <ExternalSourceMapping source-id="LDAP" name-in-source="mail" />
  </AttributeDefinition>
        <AttributeDefinition attribute-name="First Name" include-in-filter="false" include-in-list="false">
    <ExternalSourceMapping source-id="LDAP" name-in-source="givenName" />
  </AttributeDefinition>
        <AttributeDefinition attribute-name="Last Name" include-in-filter="false" include-in-list="false">
    <ExternalSourceMapping source-id="LDAP" name-in-source="sn" />
  </AttributeDefinition>
  <AttributeDefinition attribute-name="Phone" include-in-filter="false" include-in-list="true"></AttributeDefinition>
  <AttributeDefinition attribute-name="Fax" include-in-filter="false" include-in-list="false"></AttributeDefinition>
  <AttributeDefinition attribute-name="Cell" include-in-filter="false" include-in-list="false"></AttributeDefinition>
</ContactAttributeDefinitions>

      Each AttributeDefinition element defines a user attribute that may be displayed (through the attribute-name XML attribute). The "Full Name" and "Email" attributes are the only ones required.

      If the include-in-filter XML attribute is set to true, the attribute is searchable in user create and list pages, otherwise it is not.

      The include-in-list XML attribute is currently ignored, and all attributes will show up in user detail pages.

      The ExternalSourceMapping element determines which user attributes come from external sources through the source-id XML attribute. Currently only "LDAP" is supported, which signifies that LDAP is queried, and which LDAP element is used (name-in-source attribute), for user information related to that particular user attribute. If there is no associated ExternalSourceMapping element configured, then a Usage Controller can set that particular user attribute in the library. A user can also set any such attributes for their own account.

      The sample Attribute Definition document configures 7 possible attributes for a user (Full Name, Email, First Name, Last Name, Phone, Fax, and Cell). The sample document also states where the user attribute values should come from in LDAP. In this case, the "Full Name" for a user comes from the LDAP "cn" attribute, the "Email" for a user comes from the LDAP "mail" attribute, and so on. The "Full Name" and "Email" attributes are the only searchable attributes.

      One attribute that may be helpful to include is the account name of a user. This may be configured by adding the following entry to the XML file and editing the name-in-source attribute to correspond with the attribute you are using to authenticate users through LDAP (the attribute configured in the ldap.url parameter in the lib.conf file): 

              <AttributeDefinition attribute-name="Account Name" include-in-filter="true" include-in-list="false">
    <ExternalSourceMapping source-id="LDAP" name-in-source="uid" />
  </AttributeDefinition>

    Full-text artifact search on Oracle

    By default, new Lifecycle and Portfolio Manager installations perform full-text searching of asset artifacts on Oracle. However if you've upgraded from RM 6.4, and had not yet enabled Full-text artifact searching on Oracle, perform the following steps:

    Run the following to enable all asset queries to search for search terms within asset artifacts. See also the installation property "database_text_enabled" in this document. Note that there will be changes made to the database so the JDBC database user must have full privileges during this procedure.
    1. Install Oracle Text. This may have already been installed into the Oracle instance when it was created. However if it was not you'll need to install it and ensure the default Oracle Text entities are installed.
      • The ORACLE_HOME/ctx/admin/defaults/drdeflang.sql script ensures the default stoplist, lexer, and wordlist are setup for your language. You'll need to run this script if these are not setup in the Oracle instance.
    2. On the Lifecycle Manager Server, from the Installation Admin Console:
      • Execute Command
      • Command name: EnableTextSearchOnOracle
      • Choose the desired library from the dropdown list
      • Invoke
      • Several confirmation messages are output
      • Repeat for each library where full text searching is desired
      If an error is generated such as:
      "PLS-00201: identifier 'CTX_DDL' must be declared
      ORA-06550: line 1, column 7:
      PL/SQL: Statement ignored"
      Have your DBA execute this SQL query:
      GRANT CTXAPP TO yourDBuser
      Then re-run the command above.
    3. On the database, execute the GRANT statements output from the command above. These are listed here:
        --Change "dbuser" to the database user name in your environment.
        GRANT EXECUTE ON CTX_CLS TO dbuser;
        GRANT EXECUTE ON CTX_DDL TO dbuser;
        GRANT EXECUTE ON CTX_DOC TO dbuser;
        GRANT EXECUTE ON CTX_OUTPUT TO dbuser;
        GRANT EXECUTE ON CTX_QUERY TO dbuser;
        GRANT EXECUTE ON CTX_REPORT TO dbuser;
        GRANT EXECUTE ON CTX_THES TO dbuser;
    After successful command completion, set the installation property "database_text_enabled" to true so new libraries are automatically indexed.

    Single Sign-on (SSO) Installation and Configuration

      Single Sign-on (SSO) is a login mechanism that allows authorization by an external web server plug-in. This enables you to integrate with your own "single sign on" solution.

      This capability is enabled by configuring the following property in the lib.conf. 

       userIdHeader=

      The value of this entry should specify the HTTP header name that contains the authenticated user's user id (account name). So, for example if the lib.conf specifies "XYZ_USERID", the logon logic will look for a HTTP Header with that name. Effectively, you enable external authorization by specifying non-null values for this entry.

      If incoming HTTP request contains the specified header, the login logic assumes that this user has been authenticated externally and does not present a login challenge page. Instead, it attempts to find an existing User object with the account name specified by the configured user id header. It then uses the found User object to establish a session for the user.

      When properly configured and integrated with an external authentication mechanism, users never see the login challenge. Instead they see the challenge presented by the external authentication mechanism. In this mode it is not necessary for Usage Controllers to define new users.

      When external authorization is configured, No "Logout" link appears on the top navigation bar.

      The following resources should not be protected from SSO authorization:

      • contextroot/application/access
      • contextroot/ServerInfo
      • contextroot/services
      • contextroot/servlet/BirtReportServlet
      • contextroot/servlet/WSDLResourceServlet

    Running with an alternate database user

    By default, the database schema owner is used to create and modify the database entities (tables, views, sequences, etc). If you would like to configure a different user, there will be some actions that do not work as they modify the underlying schema. These actions need to be performed as the schema owner and therefore if you have configured an alternate database user, you will see certain permission issues running them. Portfolio Manager or Lifecycle Manager must be reconfigured to use the schema owner in this case. The potentially problematic actions are:

    • Creating a library
    • Removing a library
    • Using the SetAttributeDefinitions command
    • Other commands that may modify the database schema

    If you would like to configure an alternate database user, follow these instructions.

    1. Run the GenGrants command and save the result to a file
    2. Copy the file to the database server
    3. Execute the SQL against the database
    4. Modify the configuration file and set the database.schema.name to the schema the tables are in. Remember to run ApplicationRoot/bin/install update-config
    5. Modify the JDBC resource in the application server to use the alternate database user and password
    6. Restart the application server

    Other setup notes:

    • If you are using any of the standard reports, which are available on a new library or if you run the CreateStandardReports command, then you'll have to set the database_schema_name parameter on the reporting links to the database schema owner. For example you may need to modify or append "&database_schema_name=logidex" to all the standard reports.

    Notes on Uploading Large Artifacts

    • WebLogic

      WebLogic uses /tmp/_wl_proxy to write the file while it is being uploaded. Typical problems tend to be a permissions/mode or disk full problem. Verify you have enough space in /tmp for a file at least the size you are trying to upload.

      Also verify that the /tmp/_wl_proxy directory is owned by the web server user (User and Group in the httpd.conf file). You can delete the /tmp/_wl_proxy directory and it will be recreated with the correct permissions.

    • WebSphere

      There is a configuration element in the plugin-cfg.xml web server plug-in configuration file which limits uploads to 10MB by default. To increase the limit to (for example) 30MB, change the following line in the WebSphere plugin-cfg.xml file. 

      from <ServerGroup Name="MY_NODE/Default Server">
to <ServerGroup Name="MY_NODE/Default Server" PostSizeLimit="30000000">

    Migrating Artifacts to a Long Tablespace

    You can use Long Table spaces for the columns that store binary data. Previously regular table spaces were used to store binary data which was inefficient. It is recommended to upgrade to use Long table spaces.

    Note: If you have used the advanced tablespace tab for library creation, please contact SOA Software support.

    1. Create a long tablespace
      • Oracle example:
        CREATE TABLESPACE LOGIDEXTRUELONG DATAFILE '/home/oracle/oradata/LOGIDEX/LOGIDEXTRUELONG.dbf' SIZE 20M REUSE AUTOEXTEND ON NEXT 5M MAXSIZE UNLIMITED;
        ALTER USER LOGIDEX QUOTA UNLIMITED ON LOGIDEXTRUELONG;
      • UDB example:
        CREATE LARGE TABLESPACE LOGIDEXTRUELONG MANAGED BY DATABASE USING (FILE '/home/db2inst1/LOGIDEX/LOGIDEXTRUELONG01.dbf' 65536)
        GRANT USE OF TABLESPACE LOGIDEXTRUELONG TO USER LOGIDEX
    2. Update the configuration file and set the logidex.long.ts parameter to the newly created tablespace. Also set the schema.version parameter to "1". Remember to run "update-config" and restart the application server
    3. Run the MigrateArtifacts command.
    4. Confirm that artifacts can be accessed in the library.

    Saving files in UTF-8

    One needs to be concerned with saving files that have non-ASCII characters. There are several different encodings that one can use to safely preserve extended characters (UTF-16, UTF-8, etc). Portfolio Manager and Lifecycle Manager use UTF-8 encoding in various places throughout the application. Most of these issues should be transparent to the end-user, but some should be noted. Whereever input takes the form of an XML file, the XML file has a declaration for specifying the encoding. Even if it is not specified, the default encoding is UTF-8 and the document will be interpreted as being encoded as such. It is recommended to leave this set to UTF-8 and ensure that your file is saved in UTF-8.

    Most XML files displayed to the user will be preserved as UTF-8 if they are saved using the browsers File -> Save As option. Do not copy and paste this text from the browser to a text editor unless you know the text will be preserved as UTF-8 encoded without a BOM (byte-order mark).

    Also, if overviews are uploaded using Automation Extensions, they should be encoded using UTF-8 .

    It is very easy to mess up a file by saving it in the incorrect encoding. Therefore these instructions should be used anytime you save a file containing non-ASCII characters.

    • Notepad: Choose the File menu, then Save As. Under the Encoding dropdown, select UTF-8 and save the file.
    • WordPad: WordPad does not handle UTF-8 encodings, use Notepad instead. WordPad does have a Unicode option, but unfortunately this is UTF-16, not UTF-8.
    • Vim: Some UNIX systems have an enhanced version of VI installed called Vim. Use the command ":set encoding=utf-8" when saving your file.
    • Visual Studio: Choose the File menu, then Advanced Save Option. In the Encoding dropdown, select Unicode (UTF-8 without signature) - Codepage 65001.
