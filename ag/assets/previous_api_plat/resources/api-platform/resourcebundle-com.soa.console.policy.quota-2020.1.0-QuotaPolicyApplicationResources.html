com.soa.console.policy.quota.bdw.page1.description
<p>The "Modify Bandwidth Quota Policy" screen allows you to update the bandwidth quota policy. A Bandwidth Quota is defined as the a number of kilobytes or megabytes per second that a consumer can upload or download at any given time. </p><p>If the quota is exceeded, the runtime will throttle the traffic to conform to the quota policy. The quota is also assigned to either the request (upload) or response (download). </p><p>No alerts are generated for this policy since the bandwidth consumed is a function of the network speed and capabilities of the service provider, not the consumer.</p>

com.soa.console.policy.quota.con.page1.description
<p>The "Modify Concurrency Quota Policy" screen allows you to update the concurrency quota policy. A Concurrency Quota is defined by the "Concurrency Limit" at any given time. If the quota is exceeded, the "Fault Message" will be returned to the service consumer and an alert will be logged.</p><p>You can define a fault message by specifying a name (Fault Message), SOAP fault element that holds errors and status information for a SOAP message (Fault Element Name), URL that points to the fault code schema (Fault Namespace URI), and URL that points to the fault code schema (Fault Namespace URI).</p><p>The "Time to Clear Alert" parameter defines the time that the throughput needs to be within the limit before a second alert is sent to signify that the consumer is once again within their quota.</p><p>The "Send Reminder Alert" is an alert that is sent every "Reminder Alert Interval" until the consumer is within their quota.</p><p>The "Alert Code," "Clear Alert Code," and "Reminder Alert Code" drop-downs display a list of custom alert codes that have been defined in the Alerts > Alert Codes section of the Management Console. Alerts will be sent when the concurrency limit has been exceeded (Alert Code), when the concurrency limit is no longer being exceeded (Clear Alert Code), and when the concurrency limit continues to be exceeded (Reminder Alert Code) based on the specified "Send Reminder Alert" interval.</p>

com.soa.console.policy.quota.dyn.page1.description
<p>The <strong>Modify Script Policy</strong> screen allows you to update a policy using a defined script language. The script can be written in any of the languages available in the drop-down list.</p><p>You can import an existing script by importing it to the current script definition via the <strong>Imports</strong> section. To accomplish this, in the <strong>Available Scripts</strong> section select a script from the tree, then use the left arrow to copy it to the <strong>Imported Scripts</strong> section. After the script is imported it can then be referenced in the script source (e.g., samplescript(value1,value2);).</p><p>Note that only scripts that match the language type will be available for selection in the tree. The script does not return a value.</p><p>Several predefined functions and variables are available to build a policy expression that is evaluated at runtime.</p><p>From the script workspace, to use a pre-defined method, type CTRL+Space, choose from the list, and then type a period (.) after the method.</p><p>When you are ready to save your script, click <strong>Finish</strong>.</p>

com.soa.console.policy.quota.tps.page1.description
<p>The "Modify Throughput Quota Policy" screen allows you to update the throughput quota policy. A Throughput Quota is defined by the "Throughput Limit" over a time period.</p><p>If the quota is exceeded, the "Consumer Fault Message" will be returned to the service consumer and an alert will be logged. </p><p>The "Time to Clear Alert" parameter defines the time that the throughput needs to be within the limit before a second alert is sent to signify that the consumer is once again within their quota. </p><p>The Reminder Alert is an alert that is sent every "Reminder Alert Interval" until the consumer is within their quota. </p><p>The "Queue Size" is an optional parameter that defines the number of concurrent threads that the runtime will queue up before denying access. This delays the consumer requests in an effort to overcome a brief spurt in traffic without denying access.</p>
