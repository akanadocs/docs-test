Use Case: Upgrading with Zero Downtime

This use case outlines a scenario for achieving a zero-downtime upgrade in an implementation of the Akana API Platform spanning multiple data centers.

This example includes two data centers, but the approach would work in exactly the same way for any number of data centers. The upgrade is from version 8.2 to 8.4.

Note that in this example, the term zero-downtime is used to mean that the API Gateway has zero downtime. It's possible that during the upgrade process either Policy Manager or Community Manager, or both, might experience some downtime. This might be necessary if the upgrade includes database schema changes.

Note: This use case provides an approach for manual upgrade across multiple data centers. However, the best, and recommended, approach for upgrade in a production environment is via automation, which is available as a service from the Akana Professional Services department. For details about automation options, contact your Akana representative.

Multiple data center scenario

This approach upgrades data centers one by one. While one data center is being upgraded, the remaining data centers (one or more) service all traffic.

Key points relating to this use case:

  • Only Policy Manager and Community Manager have direct access to the MongoDB database. This assumes that the Network Director is pushing all its writes to MongoDB via the Policy Manager server.
  • To maintain zero downtime, all containers must have access to the Master Policy Manager. During the upgrade process, the container that is designated as the Master Policy Manager changes between data centers. The Master Policy Manager is defined as the Policy Manager container, or set of Policy Manager containers, that are configured against the writable database.

Step 1: Upgrade Data Center B

In this step, we first move all traffic to data center A, so that traffic is uninterrupted while data center B is being upgraded. We then perform the upgrade.

High-level summary of steps

For this scenario, you'll need to complete these steps:

  1. Change the databases: define the writable database and change the direction of the replication.

    We flip the databases and make data center B the writable database, and change the direction of the replication. This allows you to move the Master data center back and forth. If you prefer a different sequence, you can start with upgrading data center A, and its database will always stay active.

  2. Update the scheduled jobs.

    Changing the Master/writable data center also requires an update to the scheduled jobs. When building the new Master Policy Manager, the scheduled jobs must be running from this container. To do this, set the property org.quartz.scheduler.enabled to true. For all other containers that have scheduled jobs installed, set the property to false. For instructions, see Step 2-1: Turn off scheduled jobs. Remember to restore the setting afterwards.

  3. Upgrade the data center B environment.
Upgrading the data center B environment

The order of execution for upgrading data center B is:

  1. Upgrade a Policy Manager container first. As part of the upgrade process for this Policy Manager container, the database is upgraded with any required schema changes.
  2. After upgrade of the initial Policy Manager container is complete, upgrade all other containers. Order doesn't matter; you could even upgrade additional containers concurrently.

Step 2: Test

Once you've upgraded a data center, you can test by conducting percentage testing of the environment. This allows you to upgrade one side of the model, and then bring the upgraded data center back into receiving traffic. You can configure the router to direct only a certain amount of traffic to the upgraded side.

The illustration below shows a 60/40 mix. This means that 60% of the traffic continues to go down the 8.2 branch, and 40% is going to the upgraded data center. This allows you to validate that changes in the upgrade are not causing any issues.

Step 3: Upgrade Data Center A

Once you're sure everything is working perfectly with data center B, it's time to upgrade data center A.

To do this, first direct all traffic from the global router to data center B. The Master Policy Manager is now the Policy Manager, or bank of Policy Managers, that are running in data center B.

The automation container properties file adds the property pm.master.address, which points to the Master Policy Manager address. Single data center scenario

Even if you only have a single data center, you can still achieve a zero-downtime deployment. The approach is very similar to the approach for multiple data centers; the only difference is that rather than a single data center remaining active, one side of the node remains active.

The database is never moved; you will always have a single database instance that is always the active database.

You don't have to worry about using a Master Policy Manager instance, since there will never be a read-only database in the model.

The connections that are important in this model are the links into the Policy Manager cluster first. The steps are:

  1. First, remove 1 and direct all traffic to a single Policy Manager.
  2. Upgrade the Policy Manager that is out of the cluster.
  3. Once you've upgraded that Policy Manager, add it back into the cluster.
  4. Now upgrade the other node.
  5. Follow the same steps for upgrading the Network Director, one at a time.

Note that this is only viable for data centers that have multiple nodes deployed; at minimum, the gateways must have multiple nodes. Zero-downtime upgrades can still be achieved with single Policy Manager node clusters.
